AriaCoda

AriaCoda is a fork of the ARIA open source (GPL) library (from Omron/Adept MobileRobots). Some of the many improvements made over the original ARIA code include analyzing and testing the code to find and fix bugs, modernizing parts of the C++ codebase (still conforms to C++14, however, for compatibility with old compilers sometimes required by users) and refactoring buggy or unsafe implementations of API functions, optimizing a key LIDAR data processing component, removing obsolete code, overall optimization (especially reducing memory allocation and copying). All warnings and errors revealed by adding extra compilation warning flags and enabling address and UB sanitizers have now been resolved (plus issues indicated by tools such as clang-tidy and cppcheck). Code that was only used internally by Adept/Omron was removed. (The ARIA library was used both internally for Omron Automation's industral mobile robot products, and provided directly to research customers as a development SDK.) A simple continuous integration job was added to perform test builds with various compilers and platforms. (ARIA has never had automatable unit tests; I have added a few but not many yet.) Some bug fixes and improvements have been done to support users previously using ARIA or who are using ROS with MobileRobots hardware platforms.

The git changelog for AriaCoda can be found here.

The AriaCoda Github page can be found here.

AMRISim

AMRISim is a fork of the open source (GPL) MobileSim mobile robot simulator (from Omron/Adept MobileRobots). I have made a couple of improvements such as adding infrastructure to support additional client interfaces and adding a new ROS interface in addition to the existing Pioneer/ARIA interface (this allows users to use ARIA, AriaCoda, or ROS1 libraries in their own application code to easily choose, at runtime, to connect that application code to either a real robot or AMRISim via either ARIA, AriaCoda, or ROS), modernizing and optimizing parts of the C++ code (most of it is still C++11 style, and still retains the old Omron/Adept MobileRobots coding style), analyzing, testing and fixing bugs (both in AMRISim itself and in C code in the "Stage" simulation engine it is built around). Upgrading from GTK2.x to GTK4 was also planned but not yet started.

The git changelog for AMRISim can be found here.

Map Viewer

Unreleased, In Development

Map and nautical chart viewer with GPS tracking, implemented in C++ and QML/Javascript using Qt 6.x for both Mobile and Desktop use. Intended to be simple and easy to use while hiking, sailing, boating, or for small scale map annotation or geolocated field data collection etc.

Tools used include C++20, CMake, Conan, GDAL, catch2, Qt6, QtCreator, git, Android Studio, Android NDK

Source code is not public, but may be viewed upon request.

Implemented:
  • Load and display maps and charts in various raster file formats including GeoTIFF or BSB (KAP) files (temporary implementation for testing).
  • Read GPS data and automatically move map image to track position (current location is shown near bottom of view; map moves relative to screen and this position as you move).
  • Temporary zoom in/out or set zoom.
  • Initial (rudimentary) course over ground calculation using GPS position history. Automatically rotate map view to orient to current course over ground, show course line and bearing.
  • Automatically determine magnetic compass declination (magnetic variance).
  • Select destination waypoint from list with one click. Highlight destination waypoint on map, display bearing line to destination, angle and distance.
  • Load waypoints from GPX (XML format) file.
  • Convert and display headings and bearings in degrees relative to both true and magnetic north.
  • Display various data (from GPS/mobile device sensors, and calculated by app) in resizable grid view.
  • Partial test coverage of some data and navigation calculations, geographic projection, data logging and other utility classes.
  • Runs on Android and Linux Desktop.
  • Optionally record all data received and calculated to a CSV data log file.
In Progress/Next:
  • Better course-over-ground calculation with reusable position history utility.
  • Use QtDesigner to design a settings UI page (and add more features to waypoints list page.)
  • Additional tests with more code coverage.
  • Windows Desktop version.
  • Additional data calculations such as VMG, True Wind, etc.
  • Use computed true wind direction to compute sailing laylines to waypoint and visualize on map view.
  • Read GPS data from network (wifi/ethernet) and Bluetooth interfaces.
  • Support additional map file formats (GeoPDF, GeoPNG, GeoJPEG, KML/KMZ with overlay, etc.) via GDAL.
  • Record past track as you move. Save to GPX (XML format) file. Include positions and any other data selected.
  • Improve UI elements: choosing map file(s), viewing/editing waypoints, choosing preferences etc.
  • Nice visualization of GPS position accuracy.
Future Ideas:
  • Tools to follow route (a sequence of waypoints)
  • iOS and MacOS versions.
  • "Bare" embedded version for OEM devices. (QNX, VxWorks and other platforms supported by Qt are also possible.)
  • Split screen and other additional features on larger size devices (Desktop, tablets.)
  • Sync data between devices (e.g. Desktop to mobile.)
  • Visualization of wind, tide and current, manual approach/guide lines related to specific points.
  • Display AIS data if available via NMEA
  • Other features useful for sail boat/yacht racing and cruising.
  • Support for vector map data.
  • Simplified visualization of elevation, slope and other terrain information
  • Data collection tools such as photos, interconnection with other apps, field notes, etc.
  • ... plus many more ideas.

With Omron/Adept MobileRobots

MobileRobots was a business unit of Omron Automation (previously owned by Adept Technology) specializing in providing hardware and software platforms used for education and advanced research in mobile robotics. An archive of the website featuring photos and information on the products is availabe via The Internet Archive. Customizable systems were sold to research universities, institutes, government agencies and private enterprises around the world, and provided with comprehensive software libraries and highly regarded personalized customer support. Omron Industrial Automation closed this business unit and discontinued support for all its products in 2018.

ARIA

ARIA was the core library used to integrate mobile robotic systems and implement applications with Omron/Adept MobileRobots platforms. It was used internally to help implement the core robotics application (configurable autonomous mobile robot motion and behavior.) and provided to R&D customers as an SDK to help develop their own applications. It provided a C++ API for controlling the mobile robot and its components, and for receiving data from the mobile robot and various sensors. It also provided utilities and building blocks generally useful in robotics.

Development of ARIA was begun around 2003, and so over the years retained many legacy attributes of older C++ code. Later, some of this (but not all) was improved upon in my AriaCoda fork (see above).

MobileSim

MobileSim was an abstract simulator for multiple mobile robots in a two dimensional environment. Simulated sensors returned realistic sensor data based on the simulated robot's position with respect to walls and other obstacles defined in the environment, and the robot moved in the environment when commanded similar to the real robot (using parameters provided using the same configuration files used in ARIA).

MobileSim included its own implementation of the same interface protocol that applications (via ARIA) used to communicate with a real robot platform. ARIA automatically detects whether MobileSim (or later, AMRISim) is running and simply uses the simulator's TCP port instead of the real robot's serial port. MobileSim included my own "clean room" re-implementation of the interface, since it was released as open source code, with compatibility verified through testing. MobileSim also extended this protocol with additional features.

See also my AMRISim fork of MobileSim, described above.

ROS and Gazebo support

ROS (Robot Operating System) is an open source project that around 2010 began to emerge as a popular framework for robotic software development in academia, especially for mobile robots. Therefore it eventually became an important ecosystem to support and participate in. (This is something I and some other engineers identified and began discussions about within MobileRobots.) I did this by partnering with customers at the University of Zagreb, Croatia, who had already begun developing a ROS package to support our products. I improved the documentation, including a detailed tutorial, and contributed some fixed and feature improvements; later, when the original maintainers had moved on, I took over maintainence of the package, managing contributions from some users and answering questions. One project performed by summer college interns whom I supervised, was to test and improve our ROS support with features, example code, as they designed and constructed their own ROS-based application.

I also created a package of configuration support which could be used by MobileRobots product users, including 3D graphical models and physical parameter definitions (URDF files) of the various MobileRobots products which could be used by other ROS components and by the Gazebo simulator (which features 3D space and game-like physical interaction simulation.) I did this by combining originar engineering CAD information, my own design, and physical testing to verify various parameters.

Finally, I also created a ROS component that provided a high level API for interacting with our robust, proprietary, autonomous nagivation implementation, so that users could incorporate that into their ROS-based system design as well.

Misc Device Support

As part of our highly customizable mobile robot products, we would frequently integrate third party devices such as sensors and actuators. I was responsible, along with an electrical/mechanical systems integration engineer, for evaluation of potential third party integrations, and if a product was chosen for integration, determine the appropriate level (i.e. my time and effort) for initial software support, and plan for any future development. (Sometimes, only a few third party device units would end up sold, while some would become popular and frequent additions to a customer system.)

For some devices, I created a C++ wrapper library with improved, easy to use API around the vendor-supplied C library or example code, or around other available open source libraries or Linux driver interfaces (with instructions on how to install and configure required Linux and/or Windows drivers, libraries, etc.). For others, I simply implemented the device communications protocol according to documentation/specification, and included it in ARIA (e.g. GPS receivers via NMEA protocol) or in a separate support library. (When similar ARIA integrations were done by other Adept/MobileRobots developers, I would sometimes contribute documentation, examples and tests.) Or sometimes, I created a fork of a third party open source library with minor fixes, example code, documentation, or improvements for easier use by our customers.

A few examples using open source code can be found at https://github.com/MobileRobots. Some others were not released as open source code and are no longer available.

Demos

I would occasionally create a product demo to display at trade conferences. The most recent demo featured a robot with two manipulator arms, and a Microsoft XBox Kinect sensor mounted on a pan/tilt unit. In the demo, the mobile robot would autonomously drive a route around the exhibition floor (avoiding people and other obstacles) using Omron/Adept's autonomous navigation implementation. When it returned to our table, it would stop and move the manipulator arms (in a predetermined sequence). Monitoring software running on a laptop (connected to the robot over wifi) showed the position of the robot, and visualization of sensor data including point cloud of obstacles detected by lidar, filtered point cloud derived by processing the Kinect depth data, and remotely fetched video stream showing RGB and depthmap data from the Kinect as images (using OpenCV to convert the image formats). Remote control, configuration and monitoring of system performance data was also possible. (Some of the code used in this demo can be found at pioneer-lx-manipulator-demo and KinectArVideoServer. This was implemented in the robot as a distributed system on two computers, one lower power computer performing autonomous control of the robot, and a second computer with a basic GPU capable of doing some of the Kinect depth map processing, and control of the manipulator arms.

This robot configuration was sold as a special package, in which I assisted in overall design, as well as putting together documentation and example configuration (e.g. Linux network configuration, example ROS configuration, etc.)
